# Best Practices

## 1. 明确“执行范围”和“展示目录册”是两类状态

这是本次问题的根因。

- 执行范围：
  - 本次任务的 `roots/completedRoots/activeRoot`
  - 用于进度条与运行态
- 展示目录册：
  - 历史根目录详情 + 手工拉取的新目录
  - 用于 toggle 选择与异常目录定位

不要再让同一组字段同时承担两种职责，否则 scoped run 一定会继续产生覆盖问题。

## 2. “拉取目录详情”必须是幂等、部分成功、可重试

- 批量输入时允许部分成功
- 已存在目录再次拉取应更新名称/估计值，不报冲突
- 单个目录失败不应清空整个列表

这能显著降低管理员排障时的心智负担。

## 3. 前后端契约先行（OpenAPI）

- 先改 `api/openapi.yaml`
- 再生成：
  - `api/types.gen.go`
  - `web/src/api/generated/*`
- 最后实现 handler / hook / UI

避免前端先写临时字段，后续再回补契约。

## 4. 双层防线处理危险组合（force_rebuild + scoped）

建议同时做：

- 前端防线：直接禁用或阻止提交
- 后端防线：返回 `400 BAD_REQUEST`

原因：

- 前端防线提升体验
- 后端防线保证 API 被脚本调用时也安全

## 5. UI 反馈要区分三类动作

- 拉取目录详情（inspect）
- 启动同步（start sync）
- 同步轮询（polling）

三者应有独立 loading/error 状态，避免：

- 拉取目录详情失败导致同步状态被误显示为失败
- 轮询错误覆盖 inspect 错误消息

## 6. 保持向后兼容的渐进发布策略

- 新前端优先读取 `catalog*`，缺失时回退 `rootProgress`
- 新后端可先支持新字段，但不强制前端立即使用
- 先加测试，再改 UI，最后移除旧文案/逻辑（若需要）

## 7. 测试优先覆盖真实痛点

本次最关键回归用例不是“能提交请求”，而是：

- 局部同步后列表不被覆盖
- 拉取目录详情不会触发同步
- running 状态下不会产生并发操作

这些用例直接对应用户反馈，应作为回归测试的第一优先级。
